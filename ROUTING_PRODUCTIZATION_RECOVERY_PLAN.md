# Aegos 分流页产品化纠偏计划

本文件现在是 `PRODUCT_MATURITY_RECOVERY_PLAN.md` 的子计划，只覆盖 3.4.16 分流页产品化。它不能单独代表 Aegos 产品成熟，也不能替代首页、节点、连接、订阅、诊断、日志、设置、安全和安装的全软件验收。

本文件是 3.4.10 之后的纠偏计划。它的目的不是给已经完成的工作找理由，而是承认当前分流页只达到“工程地基可见”，还没有达到“普通用户可用”。

## 现状判定

当前分流页不能视为完成。

- 工程层面：已经有只读快照、策略组、规则解析、草稿预览、冲突提示、基础安全审计。
- 产品层面：页面像调试台，信息堆叠，普通用户不知道先看哪里、能做什么、做了会不会影响连接。
- 交互层面：草稿、验证、应用、撤销之间的关系不够清楚，页面没有形成完整任务流。
- 验收层面：已有审计偏工程安全，缺少真实用户任务验收。

因此 3.4.10 只能标记为 IPv6/DNS 地基完成版，不能标记为分流成熟版。3.5.x 必须在 3.4.20 通过后再继续。

## 新完成标准

以后分流页必须同时满足三层标准。

### 1. 工程完成

- 后端快照、规则解析、目标校验、预检、回滚、诊断接口存在。
- 不泄露订阅 token、节点密码、UUID、本地路径、真实敏感 IP。
- 不使用危险 `innerHTML` 渲染用户输入。
- 写配置前有预检，写配置时原子替换，失败能恢复。

### 2. 产品可用

- 普通用户能看懂当前规则大概怎么走。
- 普通用户能通过自然语言入口创建分流草稿。
- 用户能在应用前看懂影响范围、冲突和风险。
- 用户必须确认后才允许写配置。
- 用户能撤销自己的改动。
- 错误提示必须告诉用户原因和下一步动作。

### 3. 成熟可交付

- 快速切页、输入、生成草稿、验证、应用、滚动规则列表不卡顿。
- 小窗口、高窗口、窄窗口布局不挤、不遮挡、不跳动。
- 分流页不是调试页，而是按用户任务组织。
- 真实场景 smoke 覆盖：网站走代理、网站直连、应用走代理、地区直连、流媒体走指定策略、冲突处理、撤销恢复。
- 安装包候选版本必须通过分流页专门验收，不允许只用 release audit 代替。

## 版本路线

### 3.4.11：真实完成度审计

目标：停止继续往后冲版本，先把分流页所有缺口写清楚。

必须完成：
- 对分流页当前 DOM、状态、后端命令、审计脚本做一次清单审计。
- 输出“工程完成 / 产品可用 / 成熟可交付”三列差距表。
- 建立 `tools/routing-product-audit.js`，检查分流页是否仍停留在只读预览半成品。
- 不修改运行逻辑，只建立真实验收标尺。

验收：
- 有缺口清单。
- 有产品任务流图。
- 有分流页专用审计脚本。

### 3.4.12：信息架构重排

目标：把分流页从调试堆叠改成用户任务页面。

必须完成：
- 顶部只保留当前模式、当前默认策略、规则健康、最近命中。
- 主区域分为：当前如何走、我要新增规则、草稿与验证、规则明细、风险诊断。
- 策略组列表和规则列表不再同时大面积抢主视觉，默认折叠到“高级查看”。
- 文案必须短、清楚，避免工程术语堆叠。

验收：
- 第一屏能看懂页面作用。
- 不需要滚动就能完成新增草稿的第一步。
- 小窗口不遮挡、不挤压。

### 3.4.13：普通用户场景向导

目标：让用户不懂 YAML 也能表达分流需求。

必须完成：
- 网站：某个网站走代理 / 直连 / 拒绝。
- 应用：某个软件走代理 / 直连。
- 地区：中国大陆直连、国外网站代理。
- 常见服务：Telegram、Netflix、YouTube、OpenAI、游戏平台。
- 每个场景都生成清楚的自然语言预览和结构化草稿。

验收：
- 用户不用看规则文本也能判断草稿含义。
- 高级规则文本只作为展开信息，不作为主说明。

### 3.4.14：草稿中心产品化

目标：草稿不是一行规则，而是可理解、可处理的待办项。

必须完成：
- 草稿显示：来源、想达到的效果、目标策略、影响范围、冲突状态、验证状态。
- 每条草稿有验证、编辑、删除。
- 多条草稿有批量验证、清空、准备应用。
- 冲突不是只显示警告，要说明“已有规则会覆盖 / 新规则会覆盖 / 顺序不生效”。

验收：
- 用户能知道草稿还没生效。
- 用户能知道下一步是验证还是应用。

### 3.4.15：应用前预检闭环

目标：写配置前把风险拦在外面。

必须完成：
- 预检目标策略是否存在。
- 预检规则顺序是否合理。
- 预检是否和自动策略组、系统规则、已有规则冲突。
- 预检配置能否生成、能否被核心接受。
- 生成回滚点，但不写入生效配置。

验收：
- 预检失败不能写配置。
- 预检报告可读，不暴露敏感信息。

### 3.4.16：安全应用与回滚

目标：用户确认后才真正修改配置，并且失败能恢复。

必须完成：
- 新增后端写入命令，但必须受预检、路径限制、原子写入、回滚保护约束。
- 应用中不能锁死导航。
- 应用失败自动回滚到上一份可用配置。
- 应用成功后刷新快照，显示新规则。

验收：
- 模拟失败时配置不损坏。
- 当前连接不被无故断开。
- 用户能撤销最近一次 Aegos 创建的分流改动。

### 3.4.17：生效验证

目标：用户能知道规则到底有没有按预期工作。

必须完成：
- 验证网站命中走向。
- 验证应用规则是否可被核心识别。
- 显示：已走代理、已直连、未命中、被其他规则覆盖、目标策略缺失。
- 验证过程不改变节点、不切换模式、不接管额外流量。

验收：
- 验证结果有原因和建议动作。
- 验证失败不会让草稿永远卡在验证中。

### 3.4.18：诊断与错误解释

目标：出问题时用户知道怎么办。

必须完成：
- 规则目标缺失：提示选择现有策略组。
- 顺序问题：提示规则被更早规则覆盖。
- 自动策略组：提示核心可能自动选择节点。
- 订阅切换后规则失效：提示目标策略不存在或名称变化。
- 高级规则不支持：提示暂不支持图形化编辑，但可只读查看。

验收：
- 每个错误都有“原因 / 影响 / 建议动作”。
- 诊断不阻塞导航。

### 3.4.19：交互与性能验收

目标：分流页不能成为新的卡顿源。

必须完成：
- 规则列表虚拟化或分段渲染。
- 输入、切 tab、生成草稿、验证、刷新都不能阻塞 UI。
- 页面缓存和后台刷新与首页、节点页一致。
- 小窗口、高窗口、窄窗口布局稳定。

验收：
- 快速切换首页、节点、分流、日志、设置不卡住。
- 大规则列表滚动不掉帧到明显卡顿。

### 3.4.20：分流页产品化验收

目标：分流页达到可交给普通用户测试的水平。

必须完成：
- 分流页专用 smoke 覆盖所有关键任务。
- 安全审计、响应性审计、配置回滚审计、安装包审计通过。
- Release note 明确：分流页从地基预览进入产品可用候选。
- 打包安装包给用户测试。

验收：
- 普通用户能独立完成“新增规则 -> 验证 -> 应用 -> 查看结果 -> 撤销”。
- 不出现“页面看起来完成，实际无法使用”的情况。

## 执行纪律

- 不能跳过 3.4.11 到 3.4.20。
- 每个版本必须有明确产物，不能只改版本号和 release note。
- 每个版本必须回答：这个改动让普通用户更能完成哪个任务。
- 不能把审计脚本通过当成产品完成。
- 分流页没有达到 3.4.20 前，不继续 3.5.x。
- 如果发现规划仍不够，先修规划，再动代码。
