# Aegos 3.6.32 性能病根与优化方案

## 结论

本轮确认，Aegos 的卡顿不是单一 CSS 或单一接口造成，而是前端重复渲染、首次页面物化、后台轮询碰撞、后端大锁和测试盲区叠加后的结果。

“点击回调很快”不代表用户看到的页面已经完成。后续验收必须同时覆盖：命令响应、首个可用内容、连续帧间隔、布局位移、后台任务并发和真实 WebView2。

## 已确认并处理的病根

1. 隐藏页面重复重绘
   - 状态心跳曾重绘设置、节点组、订阅、日志等隐藏页面。
   - 现在只更新可见页面；首页时钟、流量和活跃连接也只在首页可见时写入 DOM。

2. 规则页冷启动人为延迟
   - 旧逻辑在首页初始化后额外等待 650 ms 才预取规则快照。
   - 现在活动订阅状态一就绪就立即预取，节点加载仍并行。
   - 冷路径测试会在软件启动后立即打开规则页，不再等待 1.2 秒后测热缓存。

3. 诊断页显示后再改布局
   - 缓存诊断结果曾在页面显示 72 ms 后写入，导致筛选栏和结果区跳动。
   - 现在先准备缓存布局，再激活页面。

4. 频繁 DOM 重建
   - 后台任务曾逐条合并、逐条重建任务列表。
   - 节点组、订阅菜单和日志曾在不可见时重建。
   - 现在批量合并、签名去重，并限制为可见表面更新。

5. 大面积实时毛玻璃合成
   - 静态大面板曾持续使用 12-18 px backdrop blur。
   - 现在静态面板使用预混合表面，模糊仅保留给短暂浮层。

6. 状态采集持有全局 CoreManager 锁
   - controller 流量请求和 LAN IP PowerShell 回退曾在全局锁内执行。
   - 现在先短锁取得观察参数，锁外执行外部读取，再短锁合并结果。
   - LAN IP 失败结果缓存 45 秒，避免持续触发 PowerShell。

7. 性能测试覆盖错误
   - 旧测试主要统计点击处理时间，没有测帧间隔、布局位移、GPU 合成和启动后立即打开规则页。
   - 现在增加窗口化 GPU、真实节奏切页、页面对归因、冷规则页和三轮波动门禁。
   - 性能测试必须串行执行。与 UI/交互测试并行会产生测试机资源竞争，不能作为产品回归结论。

## 当前证据

串行三轮冷启动：

- 状态可用：中位 135 ms，最坏 141 ms。
- 首页真实节点：中位 205 ms，最坏 207 ms。
- 启动后立即打开规则页：中位 201 ms，最坏 218 ms。
- 真实节奏切页 P95：16.8 ms，最坏单帧 33.4 ms。
- 三轮非预期布局位移：0。

窗口化 Chrome 在当前测试主机上被合成器限制在约 30 FPS，所有页面对都稳定在约 33.4 ms，没有规则页或节点页独有的异常峰值。该数据只用于发现页面间回归，不能代替真实 Tauri/WebView2 的 60 FPS 结论。

## 尚未消除的结构性风险

1. CoreManager 仍是大粒度互斥锁
   - 核心启动、待命核心准备、订阅部署、配置回滚等必要事务仍会持锁。
   - 同时到来的只读状态或节点请求可能排队，虽然前端导航不会被锁死。

2. `src/app.js` 仍是大型单体
   - 页面状态、任务调度、渲染和事件处理边界仍不够清晰。
   - 继续局部加条件会重新形成补丁堆。

3. CSS 历史层较多
   - 同一组件存在多轮覆盖，最终样式依赖文件顺序。
   - 这会增加样式重算成本，也提高不同窗口尺寸下回归的概率。

4. 后台状态仍以轮询为主
   - 状态、任务、活跃连接有不同周期，长时间运行时仍可能发生同帧碰撞。

5. 自动化主要运行在 Chrome fixture
   - 它能证明前端调度和 DOM 上限，不能证明真实 WebView2、显卡驱动、Wintun 和杀毒软件环境下的帧率。

## 后续实施路线

### P0：真实 WebView2 可观测性

- 在开发构建记录导航到首帧、首个可用内容、长帧、布局位移和 Tauri command 时延。
- 为每个后台任务记录排队、执行、回写和被丢弃时间。
- 使用 Windows Performance Recorder/Analyzer 或 PresentMon 对实际安装包采样。
- 日志默认聚合，不记录订阅 token、节点密码或完整地址。

验收：能明确回答一次卡顿发生在 JS、布局、合成、IPC、CoreManager 排队还是外部进程。

### P1：后端读写边界拆分

- 保留网络变更事务串行队列，不把连接、TUN、防火墙和配置部署改成无约束并发。
- 把只读 runtime snapshot、任务状态、测速状态移出 CoreManager 大锁。
- CoreController 保持可克隆，只读 controller 请求必须在锁外执行。
- 为状态提供独立快照缓存；核心事务忙时返回带时间戳的最近真实快照，不等待大锁，也不伪造断开状态。

验收：核心启动、订阅更新或测速准备期间，状态查询 P95 小于 30 ms，导航不受影响，状态最多短暂陈旧但不能错误。

### P2：前端渲染模型收口

- 把 App Shell、首页、节点、规则、诊断、设置拆成明确的 view renderer。
- 使用按字段订阅的 store selector 和 dirty flag，删除全量 `renderStatus` 扇出。
- 统一可见页调度器：同步切页、下一帧轻量对齐、空闲时加载重数据。
- 节点、规则、日志统一采用有上限的窗口化或分页模型。

验收：后台状态每秒更新时，不可见页面 DOM mutation 为 0；重复快照 DOM mutation 为 0。

### P3：视觉合成与 CSS 收口

- 把最终有效样式合并为组件层，删除被后续规则覆盖的历史声明。
- 只允许 transform/opacity 用于高频动效；列表和大面板禁止动画 box-shadow/filter/backdrop-filter。
- 为会变化的数字、IP、状态文案预留稳定宽高。
- 页面切换不做整页透明度动画，只允许局部 120-180 ms 状态反馈。

验收：真实 WebView2 常用设备切页 P95 不超过 20 ms，单帧不超过 50 ms，CLS 不超过 0.02，无文字或卡片抖动。

### P4：真实设备矩阵

- Windows 10/11，100%-200% DPI，集显/独显，WebView2 不同稳定版。
- 1 个、100 个、1000 个、8000 个节点。
- 空闲、测速、诊断、订阅更新、断网保护、TUN 和其他代理软件共存场景。
- 每个组合至少冷启动三次，记录中位数和最坏值。

验收：不以单次通过代替稳定性；任何最坏值超预算都必须有归因和风险记录。

## 硬性性能门禁

- 导航事件 P95 <= 4 ms，最大 <= 12 ms。
- 冷启动状态中位 <= 250 ms，最坏 <= 350 ms。
- 首页节点中位 <= 300 ms，最坏 <= 420 ms。
- 冷规则页中位 <= 260 ms，最坏 <= 350 ms。
- 真实节奏切页 P95 <= 35 ms，最大 <= 100 ms。
- 非预期布局位移 <= 0.02。
- 节点页可见节点 DOM <= 100，首页节点 <= 8。
- 隐藏页面不因心跳、测速或任务轮询重建。
- 测速、诊断和订阅更新不能禁用全局导航。
- 性能证据、GPU 证据、三轮证据和 soak 必须全部属于当前源码。

## 安装包说明

本轮开始时生成的包已保留为 `Aegos_3.6.32_x64-setup-baseline.exe`。包含本轮优化、供用户验证的包是 `Aegos_3.6.32_x64-setup-performance.exe`，两者使用不同 SHA-256，禁止混用测试结论。
