# Aegos 开源项目吸收开发路线

更新时间：2026-07-12

当前基准：Aegos 2.9.52

依据：

- `research/opensource-reference.md`
- `research/opensource-absorption-standard.md`
- `ROADMAP_2.9.12_TO_5.0_SECURITY.txt`
- 当前 Aegos 已有审计链：release、security、speed、responsiveness、architecture、debt、subscription、installer、soak、perf。

目标：把各开源项目拆成可吸收、可融合、可验收的路线，而不是把外部项目硬塞进 Aegos。每一项必须先通过来源核验、对比、评分、结构拆解，再进入开发。

## 1. 吸收排序结论

评分口径来自 `opensource-absorption-standard.md`：用户价值、技术适配、风险、验收可测性、时机，每项 0-3 分。进入开发最低标准为总分不少于 10，且用户价值、风险、验收可测性都不少于 2。

| 项目 | 路线 | 用户价值 | 技术适配 | 风险 | 可测性 | 时机 | 总分 | 结论 |
|---|---|---:|---:|---:|---:|---:|---:|---|
| mihomo 官方 API | A：契约吸收 | 3 | 3 | 3 | 3 | 3 | 15 | 立即吸收，作为核心契约 |
| FlClash | B：策略重写 | 3 | 2 | 2 | 3 | 3 | 13 | 立即对标，吸收交互和测速体感 |
| Clash Verge Rev | B/C：桌面工程与平台适配 | 2 | 2 | 2 | 2 | 2 | 10 | 近期吸收工程边界，不复制代码 |
| MetaCubeXD | B：信息组织重写 | 2 | 3 | 3 | 3 | 2 | 13 | 3.1 起吸收高级信息页结构 |
| v2rayN | B/C：协议兼容与迁移 | 3 | 2 | 2 | 2 | 2 | 11 | 3.3 起吸收协议矩阵和迁移经验 |
| Hiddify App | B：普通用户产品策略 | 3 | 2 | 2 | 2 | 2 | 11 | 3.2 起吸收分流向导表达 |
| sing-box | C：CoreAdapter 与网络模型 | 2 | 2 | 2 | 2 | 1 | 9 | 先研究，3.4 后进入实验，不进 3.0 主线 |
| NekoBox/NekoRay | B：手动节点字段经验 | 2 | 2 | 2 | 2 | 2 | 10 | 3.3 起吸收字段和 URI 边界 |
| NekoBox Android | B：协议覆盖清单 | 1 | 1 | 2 | 1 | 1 | 6 | 只记录，不进入近期开发 |
| jinqians/snell.sh | E：只记录模板思路 | 1 | 1 | 2 | 1 | 1 | 6 | 不进客户端主线，只做后续 Snell 参数参考 |
| clash-nyanpasu | B/C：Tauri GUI 备选参考 | 1 | 2 | 2 | 1 | 1 | 7 | 暂不进入近期，只作为后续补充研究 |

当前执行结论：

- 2.9.53 到 3.0 只吸收 P0：mihomo、FlClash、Clash Verge Rev 的稳定性、契约和对标能力。
- 3.1 到 3.3 吸收 P1：MetaCubeXD、Hiddify、v2rayN、NekoBox 的分流、连接页、协议兼容和手动节点能力。
- 3.4 以后再进入 P2：IPv6 自动、防泄漏、CoreAdapter、sing-box 实验。
- P3/P4 只记录，不允许打断当前主线。

## 2. 吸收总路线

### 2.9.53 - 2.9.58：开源吸收准备层

目标：不急着加大功能，先把核心 API 契约、对标基线、测速边界、发布回归和文案债务补齐。

| 版本 | 参考项目 | 吸收内容 | 融合方式 | 验收 |
|---|---|---|---|---|
| 2.9.53 | mihomo | 核心 API 契约 | 新建 `core-api-contract.md`，标明每个 API 是否会改状态、是否阻塞、超时、错误类型 | `audit:opensource` 加契约存在检查；`audit:speed` 验证测速不切节点 |
| 2.9.54 | FlClash | 对标测试方法 | 建立同订阅、同目标 URL、同 TUN 状态的测速/切页/切订阅对标表 | 输出对标报告；记录成功数、失败原因、首批返回时间、UI 卡顿 |
| 2.9.55 | mihomo + FlClash | 测速目标统一 | 批量、单节点、当前节点测速共用目标 URL、expected、超时和失败分类 | 首页与节点页结果同步；失败不显示“未测速”，显示原因 |
| 2.9.56 | mihomo | provider healthcheck 研究 | provider 层健康检查作为订阅健康，不等同用户切节点 | 证明 healthcheck 不改变当前节点；异常可取消、可超时、可清理 |
| 2.9.57 | Clash Verge Rev | 桌面发布回归 | 固化安装、权限、端口、WebView2、恢复动作 checklist | `audit:installer` 和 `audit:release` 必须覆盖 |
| 2.9.58 | FlClash + Aegos 本地 | 中文文案/编码债务 | 只建债务清单，不大改 UI | 无乱码、无术语堆积、失败提示能被普通用户理解 |

停止条件：

- 测速仍会自动切节点：停止进入 3.0。
- 诊断或测速会锁死导航：停止进入 3.0。
- 订阅 token 或节点密码出现在日志：停止进入 3.0。
- release audit、opensource audit、speed audit 任一失败：停止进入 3.0。

### 3.0.0：稳定候选门

目标：不是新增大功能，而是确认 Aegos 已经像成熟代理客户端一样“基础不犯错”。

验收：

- 连接、断开、重连、系统代理、TUN、断网保护状态闭环稳定。
- 一键测速、批量测速、单节点测速都不连接、不切节点。
- 切订阅、切节点、切模式时 UI 先反馈，后台失败能回滚。
- 首页、节点、连接、订阅、诊断、日志、设置快速切换不卡顿。
- 失败原因可见，日志可导出且默认脱敏。
- 安装包可在干净 Windows 设备直接安装运行，不要求用户手动装开发环境。

## 3. 3.1 - 3.3：成熟代理客户端基础能力

### 3.1.x：策略组 / 分流只读页

参考项目：mihomo、MetaCubeXD。

吸收内容：

- mihomo `/rules`、`/providers/rules`、`/groups`、`/connections` 的只读信息语义。
- MetaCubeXD 的高级信息组织思路：规则、连接、链路、日志、provider 状态。

融合方式：

- 新增“分流”或“策略”页面，但第一版只读。
- 不允许用户编辑 YAML。
- 策略组引用继续不混入普通节点列表。
- 所有 controller API 统一走 Aegos 后端封装，不能前端散调。

版本路线：

| 版本 | 任务 | 验收 |
|---|---|---|
| 3.1.0 | 分流只读页骨架 | 能显示当前模式、策略组、规则数量、命中概览 |
| 3.1.1 | 规则命中解释 | 用户能看到某个连接为什么走某个策略 |
| 3.1.2 | 连接页增强 | 显示 host、rule、chains、upload/download，可关闭连接 |
| 3.1.3 | provider 状态 | 显示订阅 provider 健康状态，不改变当前节点 |
| 3.1.4 | 诊断整合 | 诊断页检查 controller、secret、provider、规则可达性 |
| 3.1.5 | 只读分流验收 | smoke 覆盖切页、刷新、断线、无权限场景 |

### 3.2.x：普通用户分流向导

参考项目：Hiddify、mihomo、MetaCubeXD。

吸收内容：

- Hiddify 的普通用户低门槛思路：用户不写规则，也能表达“这个网站/应用走哪里”。
- mihomo 的规则语义和热重载能力。

融合方式：

- 向导生成规则，不让普通用户直接写 YAML。
- 新规则进入前必须预检、生成、热重载、验证、失败回滚。
- 规则变更必须可撤销。
- UI 只展示用户语言：网站、应用、地区、直连、代理、拒绝。

版本路线：

| 版本 | 任务 | 验收 |
|---|---|---|
| 3.2.0 | 规则数据模型 | 新增规则草稿、启用状态、来源、目标策略 |
| 3.2.1 | 网站规则向导 | 用户输入域名，选择直连/代理/拒绝，可预检 |
| 3.2.2 | 应用规则向导 | 按进程名/应用名生成规则，失败给原因 |
| 3.2.3 | 地区/关键词向导 | 面向普通用户表达，不暴露复杂 YAML |
| 3.2.4 | 热重载与回滚 | 修改规则失败自动恢复旧配置 |
| 3.2.5 | 规则审计 | audit 检查路径限制、原子写入、日志脱敏 |
| 3.2.6 | 分流向导验收 | 人工回归：添加、删除、禁用、失败回滚 |

### 3.3.x：协议兼容与手动节点成熟化

参考项目：v2rayN、NekoBox/NekoRay、jinqians/snell.sh。

吸收内容：

- v2rayN 的多协议兼容和迁移经验。
- NekoBox/NekoRay 的手动节点字段覆盖。
- snell.sh 的 Snell/ShadowTLS 参数校验思路，只吸收字段模板，不执行服务端脚本。

融合方式：

- 先建立协议能力矩阵，再补 UI 字段。
- 手动节点必须能导入、编辑、测试、连接、导出。
- 不支持的字段要明确提示，不能静默丢弃。
- 每个新增协议字段必须有 fixture。

版本路线：

| 版本 | 任务 | 验收 |
|---|---|---|
| 3.3.0 | 协议能力矩阵 | parser、mihomo、UI、测速、诊断支持状态一目了然 |
| 3.3.1 | 手动节点字段补全第一批 | Reality、uTLS、SNI、ALPN、fingerprint、flow |
| 3.3.2 | 手动节点字段补全第二批 | obfs、shadow-tls、snell、hy2、tuic 扩展参数 |
| 3.3.3 | 批量 URI 导入 | 剪贴板、多行文本、失败行原因 |
| 3.3.4 | v2rayN 迁移研究 | 只做导入方案和 fixture，不直接接多核心 |
| 3.3.5 | 手动节点验收 | 固定节点可编辑、测速、连接、导出，失败可诊断 |

## 4. 3.4 - 3.6：高级网络能力与平台工程

### 3.4.x：IPv6 / DNS / 防泄漏自动化

参考项目：sing-box、mihomo。

吸收内容：

- sing-box 的路由、DNS、TUN、IPv6 防泄漏模型。
- mihomo 当前核心可支持的 IPv6、DNS、rules 能力。

融合方式：

- 仍然以 mihomo 为默认核心。
- IPv6 模式叫“自动”，不让普通用户纠结复杂开关。
- 检测本机 IPv6、当前节点 IPv6、IPv6 出口、IPv6 泄漏。
- 节点不支持 IPv6 时自动回退 IPv4 或阻断泄漏，并提示用户。

版本路线：

| 版本 | 任务 | 验收 |
|---|---|---|
| 3.4.0 | IPv6 检测设计 | 本机、节点、出口、泄漏四类检测定义清楚 |
| 3.4.1 | IPv4/IPv6 出口检测 | 当前节点 IPv4/IPv6 能独立显示 |
| 3.4.2 | IPv6 自动回退 | 节点不支持 IPv6 时回退 IPv4，不影响连接 |
| 3.4.3 | DNS 泄漏检查 | 诊断页能给出泄漏风险和建议 |
| 3.4.4 | 防泄漏验收 | TUN/系统代理/智能分流组合测试通过 |

### 3.5.x：CoreAdapter 设计与实验

参考项目：sing-box、v2rayN、Clash Verge Rev。

吸收内容：

- sing-box 的第二核心可能性。
- v2rayN 的多核心资产管理思路。
- Clash Verge Rev 的桌面核心管理边界。

融合方式：

- 先做设计文档和接口，不默认引入第二核心。
- UI 不暴露“核心差异”，只展示用户能力。
- 实验核心不进入正式安装包，除非单独通过 release gate。

版本路线：

| 版本 | 任务 | 验收 |
|---|---|---|
| 3.5.0 | `CoreAdapter` 设计文档 | mihomo adapter 当前能力、sing-box 未来接口清楚 |
| 3.5.1 | 核心资产管理研究 | 版本、hash、路径、权限、更新策略 |
| 3.5.2 | 实验 adapter 骨架 | 不影响 mihomo 默认核心，不进入安装包 |
| 3.5.3 | 多核心风险审计 | 配置、测速、诊断、日志、安全边界全部列明 |

### 3.6.x：桌面工程成熟化

参考项目：Clash Verge Rev、clash-nyanpasu。

吸收内容：

- 托盘、后台驻留、开机启动、退出恢复。
- Windows/macOS/Linux 系统代理抽象。
- 发布通道、签名、自动更新研究。

融合方式：

- 先抽象 Windows `PlatformProxyManager`，不急着承诺 macOS/Linux。
- 托盘动作必须能恢复系统代理，不能残留网络状态。
- 自动更新必须先做签名方案，不允许裸更新。

版本路线：

| 版本 | 任务 | 验收 |
|---|---|---|
| 3.6.0 | 平台抽象文档 | Windows/macOS/Linux 代理、TUN、权限差异清楚 |
| 3.6.1 | Windows `PlatformProxyManager` | 系统代理逻辑不散落 |
| 3.6.2 | 托盘 MVP | 连接/断开、当前节点、系统代理状态、退出恢复 |
| 3.6.3 | 开机启动/后台驻留研究 | 权限、回滚、安全提示清楚 |
| 3.6.4 | 签名与更新方案 | 只出设计，不启用不安全自动更新 |

## 5. 3.7 - 4.0：成熟化、迁移与发布

### 3.7.x：性能与稳定性二次审计

参考项目：FlClash。

吸收内容：

- 体感流畅标准。
- 启动、切页、切订阅、测速、诊断、日志的无卡顿对标。

验收：

- 快速切页 p95 达标。
- 测速中所有主导航可操作。
- 大订阅列表不阻塞首页。
- 后台任务取消、超时、回滚可靠。

### 3.8.x：UI 与交互统一精修

参考项目：FlClash、Hiddify，但只吸收原则。

吸收内容：

- 普通用户优先的信息层级。
- 高级功能隐藏到高级页。
- 错误提示和下一步动作统一。

禁止：

- 不复制任何参考项目图标、样式、背景、文案。
- 不为了视觉重做核心流程。

### 3.9.x：3.x 最终候选

目标：

- 清理旧实验。
- 锁定 4.0 功能范围。
- 完成安全、发布、安装、回滚门禁。

### 4.0.0：成熟智能代理客户端

验收：

- 基础代理功能稳定。
- 分流只读、规则向导、连接页、订阅健康、手动节点成熟。
- IPv6 自动和 DNS 防泄漏具备第一版。
- 平台抽象和托盘能力不破坏 Windows 主线。
- 所有开源吸收项都有来源、评分、融合方式和验收记录。

## 7. 当前下一步

立即下一步不是做 UI，也不是加规则编辑，而是从 P0 开始：

1. `2.9.53` 建立 mihomo API 契约。
2. `2.9.54` 建立 FlClash 对标测试报告。
3. `2.9.55` 统一测速目标、返回和失败原因。
4. `2.9.56` 做 provider healthcheck 研究和安全封装。

这些完成前，不进入分流编辑、第二核心、IPv6 自动或服务端部署。

明确禁止：snell.sh 相关能力在当前路线只允许做参数模板和说明生成，不执行服务端 shell，不执行远程 root 脚本，不把服务端部署当成客户端默认能力。

## 8. 路线门禁

每个版本必须至少满足：

- 对应参考项目已在路线中标明。
- 吸收路线为 A/B/C/D/E 之一。
- 有不复制内容说明。
- 有 Aegos 融合方式。
- 有验收方式。
- 有失败停止条件。

任何版本如果没有这些信息，视为路线不合格，不进入开发。
