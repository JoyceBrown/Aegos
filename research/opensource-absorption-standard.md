# Aegos 开源项目吸收标准

更新时间：2026-07-12

适用范围：所有来自开源代理客户端、代理核心、dashboard、服务端脚本、UI 工具、测试工具的参考、移植、重写、融合与依赖引入。

目标：让 Aegos 吸收外部成熟项目时，先对比，后评估，再拆解，最后以 Aegos 自己的架构融合落地。禁止凭感觉照搬，禁止只看表面 UI，禁止把许可证、安全、性能和验收留到最后。

## 1. 总原则

### 1.1 吸收不是复制

Aegos 对开源项目的吸收分四种：

| 类型 | 是否允许 | 说明 |
|---|---:|---|
| API 语义吸收 | 允许 | 例如 mihomo 官方 API、配置字段、公开协议语义。必须写入契约文档和测试。 |
| 策略吸收 | 允许 | 例如 FlClash 的非阻塞交互、MetaCubeXD 的信息组织。必须重写成 Aegos 自己的实现。 |
| 依赖引入 | 谨慎允许 | 只允许许可证兼容、维护活跃、体积/安全可控的库，通过包管理器引入并记录风险。 |
| 代码/图标/样式直接复制 | 默认禁止 | GPL、未知许可证、视觉资产、产品文案、实现代码都不能直接复制，除非单独完成许可证审查并接受对应开源义务。 |

### 1.2 Aegos 优先级

吸收外部能力时，必须优先保护这些已经反复踩坑得到的规则：

- 测速只测延迟、推荐、失败原因，绝不切换当前节点。
- 诊断、测速、落地 IP、订阅更新都不能锁死导航和其他页面。
- 前端先反馈，后台慢慢完成；失败时回滚并给出原因。
- 核心、系统代理、TUN、断网保护、订阅切换必须事务化，失败能恢复。
- 日志和诊断必须脱敏，订阅 URL、token、节点密码不能泄露。
- 配置写入必须原子替换，路径必须限制在 app data 或明确资源目录。
- 新功能进入前必须有 audit、smoke、fixture 或人工回归清单。

## 2. 标准流程

任何外部项目进入 Aegos 任务表前，必须走完 7 步。

### 2.1 第一步：来源核验

必须记录：

- 项目 URL。
- 项目定位。
- 许可证。
- 最近维护状态。
- 是否有官方文档。
- 是否是客户端、核心、dashboard、服务端脚本、库，不能混淆类型。
- 该项目解决的问题是否正好对应 Aegos 当前痛点。

不能只写“参考某项目”。必须写清楚参考的是 API、策略、结构、交互、测试方法还是产品表达。

### 2.2 第二步：对比

对比必须用同一场景，而不是凭感觉：

| 维度 | 对比方式 |
|---|---|
| 用户体验 | 同样点击、切页、切订阅、测速，记录是否立即反馈、是否卡 UI。 |
| 网络结果 | 同订阅、同网络、同模式、同目标 URL，记录成功数、失败原因、耗时。 |
| 架构边界 | 对比状态模型、任务模型、配置生成、核心管理、错误回滚。 |
| 安全边界 | 对比 controller bind、secret、allow-lan、日志脱敏、系统权限。 |
| 可维护性 | 对比是否有清晰模块、测试、release、issue 响应。 |

输出格式：

```text
项目：
场景：
Aegos 当前结果：
参考项目结果：
差距：
证据：
是否值得吸收：
```

### 2.3 第三步：评估

每个候选能力按 0-3 分评估：

| 项 | 0 分 | 1 分 | 2 分 | 3 分 |
|---|---|---|---|---|
| 用户价值 | 基本无感 | 少数高级用户有用 | 常见用户有用 | 解决核心痛点 |
| 技术适配 | 与 Aegos 冲突 | 需要大改架构 | 能通过适配层接入 | 与现有模型自然契合 |
| 风险 | 高许可证/安全风险 | 中等风险 | 可控风险 | 风险低且可审计 |
| 验收可测性 | 无法测 | 只能人工看 | 可半自动测 | 可 audit/smoke 自动守住 |
| 时机 | 会打断主线 | 适合以后 | 可排入近期 | 当前必须做 |

进入开发的最低标准：

- 总分不少于 10。
- 用户价值至少 2。
- 风险至少 2。
- 验收可测性至少 2。
- 如果许可证是 GPL/未知，不能选择“复制代码/资产”路线。

### 2.4 第四步：拆解结构

吸收前必须拆成 Aegos 可承接的层：

| 层 | 必须回答的问题 |
|---|---|
| 数据层 | 需要新增什么字段？是否影响 profile、node、group、health、settings？ |
| 任务层 | 是前台动作、后台 job、可取消任务、定时任务，还是核心事务？ |
| 核心层 | 是否调用 mihomo API？会不会改变当前选择？是否需要热重载？ |
| UI 层 | 是否需要乐观 UI？是否会重排大列表？是否会卡导航？ |
| 错误层 | 失败如何分类？显示给用户什么？日志写什么？ |
| 安全层 | 是否触碰系统代理、防火墙、secret、订阅 token、文件写入？ |
| 测试层 | 用哪个 audit/smoke/fixture 守住？没有就先补。 |

没有完成拆解，不允许直接开写。

### 2.5 第五步：选择吸收路线

#### 路线 A：契约吸收

适合：mihomo API、协议字段、配置语义、公开标准。

做法：

- 写入 `core-api-contract.md` 或协议能力矩阵。
- 封装 Aegos 内部接口。
- 加超时、取消、错误分类、日志脱敏。
- 加 audit 验证不会误用 API。

#### 路线 B：策略重写

适合：FlClash 的交互节奏、MetaCubeXD 的信息组织、Hiddify 的普通用户向导。

做法：

- 只吸收行为原则。
- 重新设计 Aegos 数据流和组件。
- 不复制源码、图标、样式、布局细节。
- 用 Aegos 的 optimistic UI、background job、render cache 实现。

#### 路线 C：适配层融合

适合：未来 sing-box、平台代理、托盘、系统服务。

做法：

- 先定义 adapter trait/interface。
- 现有 mihomo/Windows 路线保持稳定。
- 新能力先实验，不进入默认安装包。
- 通过 feature flag 或实验分支隔离。

#### 路线 D：依赖引入

适合：小型、许可证兼容、维护活跃的工具库。

做法：

- 记录 license、体积、维护状态。
- 通过包管理器引入。
- 只放在清晰模块内，不能散落。
- 追加安全审计和升级计划。

#### 路线 E：不吸收，只记录

适合：服务端脚本、过早多核心、侵权风险高 UI、维护停止项目。

做法：

- 放入研究文档。
- 标记 P3/P4 或 blocked。
- 不进入开发任务。

### 2.6 第六步：融合实现

融合时必须遵守：

- 新代码进入已有模块边界，不能绕过 job、settings、profile、speed、diagnostics 等统一模型。
- 不能新增散落按钮补丁；按钮状态走统一 pending/busy/rollback。
- 不能新增同步重任务；长任务走后台 job 或可取消任务。
- 不能在 UI 层直接拼危险 HTML。
- 不能在 Rust 层新增非原子配置写入。
- 不能让系统级操作半成功半失败。
- 不能让测速、订阅切换、诊断互相污染状态。

### 2.7 第七步：验收

每个吸收任务完成前必须回答：

- 用户是否能感知到价值？
- 是否破坏了连接闭环？
- 是否破坏了“测速不连接”？
- 是否会导致 UI 卡顿？
- 失败是否有原因和下一步建议？
- 日志是否脱敏？
- 是否有回滚？
- 是否有 audit/smoke/fixture？
- 是否写入 release notes？
- 是否能解释“为什么这样做比原来更好”？

## 3. 当前项目吸收判定

| 项目 | 当前吸收路线 | 优先级 | 现在做什么 | 暂不做什么 |
|---|---|---:|---|---|
| mihomo | 路线 A：契约吸收 | P0 | API 契约、测速边界、provider healthcheck、规则/连接只读数据 | 不绕过 Aegos 封装直接散调用 API |
| FlClash | 路线 B：策略重写 | P0 | 非阻塞测速、快速反馈、普通用户信息层级、对标测试表 | 不复制 Flutter 代码、图标、样式 |
| Clash Verge Rev | 路线 B/C：桌面工程策略 + 平台适配层 | P0/P1 | Tauri 权限、系统代理抽象、托盘/跨平台规划 | 不直接搬 Rust/前端实现 |
| MetaCubeXD | 路线 B：信息组织重写 | P1 | 连接、规则、日志、provider 信息组织参考 | 不把 Aegos 做成复杂 dashboard |
| v2rayN | 路线 B/C：兼容矩阵 + 未来多核心抽象 | P1 | 协议能力矩阵、订阅 fixture、迁移研究 | 3.0 前不接多核心 |
| Hiddify | 路线 B：普通用户产品策略 | P1 | 分流向导、自动策略表达、TUN 用户语言 | 不引入 Flutter 架构 |
| sing-box | 路线 C：CoreAdapter 研究 | P2 | IPv6/TUN/路由模型、未来 adapter 文档 | 3.0 前不进默认核心 |
| NekoBox/NekoRay | 路线 B：协议字段与手动节点经验 | P2 | 固定节点编辑字段、URI 导入边界 | 不依赖停止维护项目 |
| snell.sh | 路线 E/P3：只记录模板思路 | P3 | Snell/ShadowTLS 参数校验和说明生成 | 不执行服务端 shell，不做 VPS 部署 |

## 4. 已开始实行的动作

本标准建立后，Aegos 的开源吸收从以下动作开始：

- `research/opensource-reference.md` 作为项目库和候选池。
- 本文件作为吸收流程、路线选择和验收标准。
- `tools/opensource-absorption-audit.js` 作为文档门禁，防止资料没有来源、没有边界、没有验收就进入主线。
- `npm run audit:opensource` 作为后续开源吸收任务的固定检查项。

下一步建议按版本推进：

- `2.9.53`：建立 `core-api-contract.md`，把 mihomo API 语义和状态副作用写死。
- `2.9.54`：建立 FlClash 对标测试表，真实记录 Aegos/FlClash 测速成功率、返回速度、UI 卡顿。
- `2.9.55`：统一测速目标 URL、expected、超时、失败原因口径。
- `2.9.56`：研究 provider healthcheck，并证明它不会改变当前节点。

## 5. 禁止清单

以下行为一律视为吸收失败：

- 看到参考项目好用，就直接复制源码。
- 看到参考项目漂亮，就高保真复制图标、背景、布局。
- 没有许可证记录就引入代码或资源。
- 没有测试就改核心网络链路。
- 没有拆解就把外部项目结构硬塞进 Aegos。
- 用“应该”“大概”“看起来”替代证据。
- 新增功能导致测速自动连接、导航卡顿、日志泄密、配置无法回滚。
- 只改表面 UI，不处理数据流、任务模型和错误模型。
