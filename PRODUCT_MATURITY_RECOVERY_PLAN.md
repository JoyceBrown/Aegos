# Aegos 全软件产品成熟度纠偏计划

本文件覆盖 3.4.11 到 3.4.20。它取代“只盯某一个页面”的做法，把 Aegos 当成完整代理客户端来验收。

细化执行表见 `PRODUCT_MATURITY_3.4.11_TO_3.4.20_DETAILED.md`。后续推进 3.4.11 到 3.4.20 时，必须同时遵守本文件和细化执行表；如果两者冲突，以细化执行表中更严格的验收条件为准。

## 纠偏结论

Aegos 不能靠某个页面、某个审计脚本、某个安装包版本证明成熟。

成熟代理客户端至少要同时满足：
- 核心能稳定连接、断开、恢复。
- 节点和测速结果真实、快速、不卡 UI。
- 订阅导入、更新、切换失败时可解释、可回滚。
- 首页让普通用户一眼看懂当前是否能上网、走哪个节点、哪里有问题。
- 分流页让普通用户能看懂、创建、验证、应用、撤销规则。
- 诊断和日志能帮助定位问题，而不是制造更多困惑。
- 设置页只暴露用户能理解的选择，高危能力有保护和解释。
- 安装包在其他 Windows 设备上能直接安装运行，缺依赖时有清楚提示。
- 安全边界清楚，不泄露 token、密码、UUID、本地路径和敏感 IP。
- 全局交互不卡顿，后台任务不锁导航。

## 软件级完成标准

### 工程完成

- 功能入口存在，后端命令存在，数据能返回。
- 有基础审计或测试。
- 不破坏安全底线。

这只能算工程完成，不能算产品完成。

### 产品可用

- 普通用户知道这个功能是干什么的。
- 用户能完成一条完整任务，不需要理解代理内核术语。
- 失败时知道原因和下一步动作。
- 操作不会让软件进入未知状态。

这是给用户测试的最低标准。

### 成熟可交付

- 主流程稳定，异常流程可恢复。
- 页面在不同窗口尺寸下不乱。
- 后台任务不影响导航、按钮和其他页面。
- 安全、性能、安装、日志脱敏和回滚门禁都通过。
- 真实任务 smoke 覆盖，而不是只检查代码字符串。

这是进入后续 3.5.x 的最低标准。

## 3.4.11：全软件真实完成度审计

目标：先停下来检查 Aegos 到底哪里只是“工程完成”，哪里真的“产品可用”。

范围：
- 首页
- 节点
- 连接
- 订阅
- 分流
- 诊断
- 日志
- 设置
- 安装包
- 安全和隐私
- 性能和响应性

必须输出：
- `PRODUCT_MATURITY_GAP_REPORT.md`
- `tools/product-maturity-audit.js`
- 每个页面的用户任务清单
- 每个页面的阻塞问题、隐藏问题、体验问题、安全问题

验收：
- 不能只写“通过”。
- 必须列出仍未成熟的功能。
- 必须明确哪些问题阻止进入 3.5.x。

## 3.4.12：核心连接闭环产品化

目标：代理软件最基本的“能不能连接”必须稳定。

必须完成：
- 连接按钮立即乐观反馈，但最终状态必须和核心真实状态一致。
- 断开连接后系统代理、TUN、断网保护、核心进程状态一致。
- TUN 关闭时，系统代理连接路径必须可用。
- 手动开启系统代理不能自动连接。
- 切换节点后自动刷新落地 IP，旧查询必须取消或过期。
- 失败时显示原因和建议动作。

验收：
- 连接、断开、重连、失败、回滚、系统代理残留检查全部通过。

## 3.4.13：节点与测速产品化

目标：节点和测速不能再成为卡顿、误连接、结果不同步的来源。

必须完成：
- 一键测速、批量测速、单节点测速绝不切换节点或连接。
- 测速结果在首页和节点页共享同一状态源。
- 失败显示具体原因：超时、DNS、TLS、认证、协议不支持、被阻断、目标不可达。
- TUIC、anytls、hysteria2、vless、trojan、ss 都有协议感知调度。
- 测速慢或测不通时，按钮和页面仍可操作。
- 直连、策略组、元数据伪节点不作为普通节点测速。

验收：
- 大量失败节点不会拖死 UI。
- 切换订阅会取消旧订阅测速任务，旧结果不能污染新订阅。

## 3.4.14：订阅产品化

目标：订阅是机场用户的入口，不能只做到能导入。

必须完成：
- 支持主流订阅格式和 URI 协议。
- 导入失败能说明是 URL、网络、格式、协议、token、服务器异常哪一类。
- 更新订阅失败不破坏旧可用配置。
- 切换订阅立即显示本地缓存节点，再后台验证。
- 切换订阅时取消旧测速、旧刷新、旧落地 IP 查询。
- 订阅可重命名，异常订阅有诊断入口。

验收：
- 四个订阅来回切换不闪、不乱、不串结果。
- 异常订阅不拖垮全局状态。

## 3.4.15：首页与常用操作产品化

目标：首页是普通用户判断软件是否可用的地方。

必须完成：
- 当前节点、落地 IP、系统代理、TUN、断网保护、实时上下行清楚显示。
- 系统代理未开启时要明确标红或提示原因。
- 常用地区、常用节点、收藏节点、固定节点逻辑清楚。
- 快捷操作不堆无用按钮。
- 首页布局在最小高度、最大高度、窄宽度下不跳动、不挤压。

验收：
- 普通用户打开首页就知道现在能不能用、哪里不对、下一步点什么。

## 3.4.16：分流页产品化

目标：分流不是调试页，是普通用户自定义规则的入口。

必须完成：
- 分流页按“当前怎么走 / 我要改什么 / 草稿 / 验证 / 应用 / 撤销 / 高级查看”组织。
- 网站、应用、地区、流媒体、AI 服务、游戏平台等场景向导可用。
- 草稿显示来源、意图、影响范围、冲突、验证结果和下一步动作。
- 用户确认后才写配置，写入必须预检、原子化、可回滚。
- 应用后能验证规则是否实际命中。

验收：
- 普通用户能独立完成新增、验证、应用、撤销一条规则。

详细子计划见 `ROUTING_PRODUCTIZATION_RECOVERY_PLAN.md`。

## 3.4.17：诊断与日志产品化

目标：诊断和日志要帮助用户，而不是卡住用户。

必须完成：
- 运行诊断时导航仍可切换。
- 诊断按钮显示进行中，但不锁死页面。
- 诊断结果按严重程度分组。
- 日志支持用户日志、调试日志、核心日志、诊断报告分层。
- 日志导出默认脱敏。
- 错误提示统一为原因、影响、建议动作、日志入口。

验收：
- 诊断中切页不卡。
- 日志导出不包含 token、密码、UUID、订阅 URL 密钥。

## 3.4.18：设置、安全与安装产品化

目标：设置页不能是危险开关仓库，安全和安装不能靠用户猜。

必须完成：
- 设置页按普通、高级、危险操作分区。
- 断网保护、TUN、系统代理、DNS、IPv6 自动模式都有清楚解释。
- controller 只绑定 127.0.0.1，默认 allow-lan 关闭。
- 端口冲突显示占用进程。
- WebView2 缺失、权限不足、防火墙失败有清楚提示。
- 安装包在干净 Windows 设备上具备依赖提示和回滚检查。

验收：
- 高危操作有确认和失败回滚。
- 安装包候选审计通过。

## 3.4.19：全局交互与性能验收

目标：Aegos 不能再“动不动卡一下”。

必须完成：
- 所有重任务进入后台任务模型。
- 页面切换立即响应，数据迟到也不能阻塞页面确认。
- 列表渲染分批、缓存或虚拟化。
- 按钮 hover、active、pending 不造成文字抖动或布局跳动。
- 测速、订阅更新、诊断、落地 IP 查询同时发生时，UI 仍可操作。

验收：
- 快速切换首页、节点、连接、订阅、分流、诊断、日志、设置无明显卡顿。
- 大量节点、大量规则、大量日志不拖垮页面。

## 3.4.20：全软件产品成熟度验收

目标：给用户安装包前，先证明 Aegos 是一个可测试的完整代理客户端。

必须完成：
- 全页面真实任务 smoke。
- 安全审计。
- 响应性审计。
- 安装包审计。
- 回滚审计。
- 日志脱敏审计。
- 真实用户路径验收记录。
- Release note 明确剩余风险。
- 生成安装包。

验收：
- 普通用户可以完成：导入订阅、连接节点、测速、切换节点、查看落地 IP、诊断问题、导出日志、调整设置、创建分流规则。
- 出错时软件能解释和恢复。
- 不再用“某个脚本通过”冒充产品完成。

## 执行纪律

- 3.4.11 到 3.4.20 不能跳过。
- 每个小版本都必须回答：这一步让哪个用户任务变得更可靠。
- 每个页面都要按工程完成、产品可用、成熟可交付三层评价。
- 任何主页面仍是半成品时，不进入 3.5.x。
- 任何重任务导致导航或按钮卡住，必须先修再继续。
- 任何安全脱敏、回滚、安装门禁失败，必须先修再继续。
